## Segment 1: The Foundation — Microsoft.Extensions.AI

Microsoft.Extensions.AI is the common language that every AI component in .NET speaks. It provides provider-agnostic abstractions for chat completion and embedding generation, so your code doesn't care whether it's talking to Azure OpenAI, Ollama, or any other provider.

### Key Concepts

- **IChatClient**: The universal interface for chat completions. Every agent, every LLM call in this app flows through IChatClient.
- **IEmbeddingGenerator**: Generates vector embeddings from text. Used by both the ingestion pipeline (writing) and semantic search (reading).
- **ChatClientBuilder**: A middleware pipeline for IChatClient, similar to ASP.NET Core's middleware. Chain behaviors like function invocation, OpenTelemetry tracing, and rate limiting.
- **AIFunctionFactory**: Creates strongly-typed tool definitions from .NET methods. Agents use these tools to interact with the world.

### Why It Matters

Without M.E.AI, every library would bring its own LLM abstraction. You'd have adapters everywhere. M.E.AI is the `ILogger` of AI — one interface, many providers, zero coupling.

### Live Demo

We register IChatClient with Azure OpenAI, wrap it with `UseFunctionInvocation()` and `UseOpenTelemetry()`, and use it to generate our first poll. The audience votes, and we analyze the results. All through IChatClient.

## Segment 2: Knowledge Engineering — DataIngestion & VectorData

Raw text is useless to AI agents. They need structured, searchable, enriched knowledge. That's what Microsoft.Extensions.DataIngestion and Microsoft.Extensions.VectorData provide — together, they form the knowledge layer.

### DataIngestion: The 5-Stage Pipeline

DataIngestion follows a clear pipeline: Reader → DocumentProcessors → Chunker → ChunkProcessors → Writer.

- **Readers** extract content from sources. `MarkdigReader` handles Markdown. Custom readers handle API responses and user input.
- **Chunkers** split documents into appropriately-sized pieces. `HeaderChunker` splits on headings. `DocumentTokenChunker` splits on token count.
- **Enrichers** (ChunkProcessors) add metadata: `SummaryEnricher` generates summaries, `KeywordEnricher` extracts keywords, `SentimentEnricher` detects sentiment.
- **Writers** persist the chunks. `VectorStoreWriter` stores them with embeddings in the vector store.

### VectorData: The Semantic Memory

VectorData provides the storage and retrieval layer. `InMemoryVectorStore` stores records with vector embeddings. `SearchAsync()` finds semantically similar content.

The key insight: DataIngestion writes to VectorData. Agents read from VectorData. It's the shared memory connecting ingestion to intelligence.

## Segment 3: Agentic AI — Microsoft Agent Framework

An agent is more than a prompt. It's an AI that can reason, use tools, and collaborate with other agents. The Microsoft Agent Framework provides the building blocks: `ChatClientAgent` for individual agents, `AgentWorkflowBuilder` for orchestration.

### Agents in Conference Pulse

- **Survey Architect**: Given a topic, searches the knowledge base for context, then generates an engaging poll question with options tailored to the audience's demonstrated knowledge level.
- **Response Analyst**: When a poll closes, analyzes the results — not just percentages, but patterns. "65% chose B, which suggests the audience already understands dependency injection but may not have hands-on experience with it."
- **Knowledge Curator**: The RAG specialist. Searches the vector store AND external sources via MCP. When it finds useful external content, it ingests it back into the vector store for future queries.

### Workflow Patterns

The Agent Framework supports multiple orchestration patterns:

- **Sequential**: Agents execute one after another. Poll generation: topic → Survey Architect → poll created.
- **Handoff**: One agent hands context to the next. Response analysis: Analyst generates insight → Curator finds supporting docs.
- **Group Chat**: Multiple agents collaborate on a shared task. Session summary: all three agents contribute their perspective to build a comprehensive summary.

The audience votes again. We run the response analysis workflow and watch the handoff between Analyst and Curator.

## Segment 4: Interoperability — Model Context Protocol (MCP)

MCP is an open protocol that lets AI systems discover and use tools from any provider. It's like a REST API, but designed for AI: tools have descriptions, parameters, and return types that LLMs can understand.

### Conference Pulse as MCP Server

Our app runs an MCP server over Streamable HTTP. Any MCP-compatible client can connect and use our tools:

- `get_active_polls` — See what polls are running
- `submit_poll_response` — Vote on a poll
- `get_poll_results` — Get results with AI-generated insights
- `search_session_knowledge` — Semantic search over everything we've ingested
- `ask_session_question` — Ask a question, get a RAG-powered answer
- `generate_session_summary` — Trigger the full summary workflow

### Conference Pulse as MCP Client

Our Knowledge Curator agent connects to external MCP servers:

- **Microsoft Learn**: Official .NET and Azure documentation. When someone asks about IChatClient, the agent fetches the authoritative docs.
- **DeepWiki**: Knowledge from GitHub repositories. Want to know how the Agent Framework works internally? DeepWiki has it.

The beautiful part: when the agent fetches external docs, it ingests them into our vector store. The knowledge base grows with every query.

### Live Demo

We connect GitHub Copilot Chat to our MCP server and ask: "What has the audience been struggling with?" Copilot calls our tools, gets live data, and answers. Then we ask the Knowledge Curator to find Microsoft Learn docs for IChatClient — watch it fetch, ingest, and now that knowledge is in our vector store forever (well, for this session).

## Segment 5: The Closer — GitHub Copilot SDK

The GitHub Copilot SDK lets you build agents that plug into the Copilot ecosystem. A `CopilotClient` connects to GitHub Copilot's LLM backend, and you can give it MCP servers to extend its capabilities.

### The Cascade

We've built a simple console app: ~15 lines of code. It creates a `CopilotClient`, points it at our MCP server, and asks: "Write a comprehensive summary of today's session."

What happens next is the entire Microsoft AI stack in action:

1. **Copilot SDK** sends the request to Copilot's LLM
2. Copilot discovers our **MCP** tools and calls `generate_session_summary`
3. Our MCP server triggers the **Agent Framework** SessionSummaryWorkflow
4. Agents search the **VectorData** store for all accumulated knowledge
5. That knowledge was put there by **DataIngestion** throughout the session
6. Every LLM call flows through **Microsoft.Extensions.AI** abstractions

One command. Six technologies. The summary streams to the terminal — and to the big screen — as the speaker narrates what's happening at each layer.
